事务

发表于 2019-04-18 | 阅读次数:

事务（Transaction）:是数据库操作的最小工作单元，是指针对数据库中的数据，进行的一些存取操作。

具有四个属性：原子性、一致性、阻离性、持久性，合起来通常被称为ACID特性:

事务的四大特性：

1 、原子性
事务是数据库的逻辑工作单位，事务中包含的各操作要么都做，要么都不做
2 、一致性
事务执行的结果必须是使数据库从一个一致性状态变到另一个一致性状态。因此当数据库只包含成功事务提交的结果时，就说数据库处于一致性状态。如果数据库系统 运行中发生故障，有些事务尚未完成就被迫中断，这些未完成事务对数据库所做的修改有一部分已写入物理数据库，这时数据库就处于一种不正确的状态，或者说是 不一致的状态。
3 、隔离性
一个事务的执行不能其它事务干扰。即一个事务内部的操作及使用的数据对其它并发事务是隔离的，并发执行的各个事务之间不能互相干扰。
4 、持续性
也称永久性，指一个事务一旦提交，它对数据库中的数据的改变就应该是永久性的。接下来的其它操作或故障不应该对其执行结果有任何影响。

Semephore和Exchanger的使用

Semaphore (信号,信号系统) 的使用

Semaphore 提供的功能是 synchronized 关键字的升级版,主要作用是控制线程并发的数量 (这一点单纯使用synchronized是完不成的 ) .

为什么要控制线程并发的数量呢?

如果不限制线程并发的数量,cpu资源很快就会被耗尽,每个线程执行任务相当缓慢,同时上下文切换也要耗时.造成系统运行效率很低.

Semaphore的同步性

多线程中同步的概念:排着队一个一个执行,有助于程序逻辑的正确性.

Semaphore的构造方法中的permit参数

参数permit代表同一时间内,设置许可的数量,即最多允许多少个线程同时执行acquire()和release()之间的代码.

参数permit大于1,不能保证线程安全性.

new Semaphore(5); 中5并不是最终的许可数量,仅仅使初始的状态值.

acquire(int permit)参数作用及动态permits许可数量

acquire(int permit)每调用一次此方法,使用x个许可.

release()同.

acquireUninterruptibly()方法的使用

作用:等待进入acquire()方法的线程,不允许被打断.

availablePermits()和drainPerimits()

availablePermits()返回此Semaphore对象中当前可用的许可数,通常用于调试,因为许可的数量实时在改变,不固定.

drainPerimits()返回立即可用的所有许可个数,并且将可用许可置为0.

getQueueLength()和hasQueuedThreads()

getQueueLength()返回等待许可的线程个数

hasQueuedThreads()判断有没有线程在等待这个许可.

公平与非公平信号量的测试

有些时候,获得许可的顺序与线程启动的顺序有关,这时信号量就要分为公平与非公平的

所谓公平信号量就是获得锁的顺序与线程启动的顺序有关,但不代表100%的获得信号量,仅仅使在概率上能得到保证.

而非公平信号量就是无关了.

在Semaphore构造函数中加入 isFair=true即可得到公平信号量运行的结果.

tryAcquire()的使用

作用使尝试获得1个许可,如果获取不到则返回false,通常与if结合,特点是无堵塞的特点,可以使线程不至于在同步处一直持续等待的状态,if不成立会执行else,程序继续往下走.

tryAcquire(int permits),尝试获得x个许可,同上.

tryAcquire(int long timeout, TimeUnit unit)在指定的时间内尝试获得许可.

tryAcquire(int permits,int long timeout,TimeUnit unit) 同上

多进路-多处理-多出路

允许多个线程同时处理任务.即每个线程都在处理自己的任务

多进程-单处理-多出路

允许多个线程同时处理任务,但执行任务的顺序确实同步的,也就是堵塞的,即单处理.

加入了ReentrantLock对象,调用它的lock方法保证同步性.

多生产者多消费者模式

类Semaphore提供了限制并发线程数的功能,此功能在默认的synchronized中是不提供的.

Exchanger的使用

Exchanger主要是可以使两个线程间互相方便进行通信.,即在两个线程之间传输数据,它比生产者/消费者模式使用的wait/notify更加方便.

exchange()方法的阻塞特性:

此方法被调用后,等待其他线程来取得数据,如果没有其他线程取得数据,则一直阻塞.

exchange(V x,long timeout,TimeUnit unit):在制定时间内没有被其他线程获取数据,出现超时异常

CountDownLatch和CyclicBarrier的使用

这两个工具类可以控制线程的同步,使线程在同步的处理上更加灵活.

CountDownLatch的使用

await()方法:判断计数是否为0,不为0,等待,为0,继续运行.

countDown()方法将计数减1,当计数减为0时,等待的线程继续运行.

new CountDownLatch(1) : 创建1个计数的CountDownLatch类的对象.

当对象执行down.await()代码时呈等待状态,程序不向下继续运行.

程序执行down.countDown()代码时计数由1变成0,呈等待状态的线程继续向下运行.

CyclicBarrier的使用

CycliceBarrier拥有CountDownLatch所具有的功能,还可以实现屏障等待的功能.它在使用上的意义在于可以循环的实现线程要一起做任务的目标,而不是像CountDownLatch一样,仅仅支持一次线程与同步点阻塞的特性.

多个线程之间,任何一个线程没有完成任务,则所有线程都必须等待.

getNumberWaiting()方法

获得有几个线程已经到达屏障点.

isBroken()方法

检查屏障是否损坏

getNumberWaiting()和getParities()方法

getNumberWaiting()作用是返回有几个线程到达屏障点.

getParties()作用是取得Parties个数.

reset()

重置屏障.线程可能会出现Broken错误.

Executor和ThreadPoolExecutor的使用

在开发服务器软件项目时,软件经常需要处理执行时间很短而数目却非常大的请求,如果为每一个请求创建一个新的线程,会导致性能上的瓶颈,因为线程的创建和销毁需要JVM频繁地进行处理,如果请求的执行时间很短,可能花在创建和销毁线程对象上的时间大于真正执行任务的时间,若这样,系统性能大幅度下降.

在jdk5中提供了线程池的概念,主要的作用是支持高并发的访问处理,并且可以将线程对象复用,核心原理即创建一个运行效率比较优异的线程池,在池中支持线程对象管理,包括创建和销毁,使用池时只需要执行具体的任务即可.

Executor接口介绍

与线程池有关的大部分类都是实现此接口的.

无界线程池:可以进行线程自动回收,无界即池中存放线程个数是理论上的Integer.MAX_VLAUE最大值.

使用newCachedThreadPoll(ThreadFactory)定制线程工厂

方法shutdown()与shutdownNow()与返回值

shutdown()方法的作用:使当前未执行完的线程继续执行,而不在添加新的任务Task,该方法不会堵塞,调用之后,线程main就马上结束了,而线程池会继续运行直到所有任务执行完才会停止.不执行shutdown()方法,线程池会一直保持下去,以便随时执行被添加的task任务.

方法shutdownNow()的作用是中断所有的任务task,并且抛出InterruptedException异常,前提是Runnable中使用if(Thread.currentThread().isInterrupted() == true)语句来判断当前线程的中断状态,而未执行的线程不再执行,也就是从执行队列中清除.如果没有if语句及抛出异常的代码,则池中正在运行的线程直到执行完毕,而未执行的线程不在执行,也从执行队列中清除.

生产者消费者模式-并发

发表于 2019-04-04 | 分类于 java ， 设计模式 | 阅读次数:

在并发编程中使用生产者和消费者模式能够解决绝大多数并发问题。该模式通过平衡生产线程和消费线程的工作能力来提高程序的整体处理数据的速度。

为什么要使用生产者和消费者模式

在线程世界里，生产者就是生产数据的线程，消费者就是消费数据的线程。在多线程开发当中，如果生产者处理速度很快，而消费者处理速度很慢，那么生产者就必须等待消费者处理完，才能继续生产数据。同样的道理，如果消费者的处理能力大于生产者，那么消费者就必须等待生产者。为了解决这个问题于是引入了生产者和消费者模式。

什么是生产者消费者模式

生产者消费者模式是通过一个容器来解决生产者和消费者的强耦合问题。生产者和消费者彼此之间不直接通讯，而通过阻塞队列来进行通讯，所以生产者生产完数据之后不用等待消费者处理，直接扔给阻塞队列，消费者不找生产者要数据，而是直接从阻塞队列里取，阻塞队列就相当于一个缓冲区，平衡了生产者和消费者的处理能力。

这个阻塞队列就是用来给生产者和消费者解耦的。纵观大多数设计模式，都会找一个第三者出来进行解耦，如工厂模式的第三者是工厂类，模板模式的第三者是模板类。在学习一些设计模式的过程中，如果先找到这个模式的第三者，能帮助我们快速熟悉一个设计模式。s
